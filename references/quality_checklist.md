# Prompt Quality Checklist

Phase 6 终检交付阶段参考此文件。
组装完整 prompt 前逐项验证。

---

## 快速自检（6 秒版）

- [ ] 我清楚这个 prompt 的目标用户和使用场景吗？
- [ ] Prompt 有明确的角色/任务定义吗？
- [ ] 有否定约束且附带理由吗？
- [ ] 有 few-shot 示例吗？
- [ ] 经过红队攻击验证吗？
- [ ] 格式适配目标 LLM 吗？

---

## 完整检查清单

### Layer 1: 结构完整性

- [ ] XML/Markdown 标签正确闭合，无孤立标签
- [ ] 所有引用的节（section）都存在，无空占位符
- [ ] 结构层次不超过 3 层嵌套（避免过深）
- [ ] 各节之间无重复定义（同一规则只在一个地方说）
- [ ] Prompt 总长度适配目标 LLM 的 system prompt 建议长度
  - Claude: 最优 < 4000 tokens，最大 ~8000
  - GPT: 最优 < 3000 tokens，最大 ~6000
  - Gemini: 可更长（长上下文），但核心指令仍应精炼
  - 通用: < 3000 tokens

### Layer 2: 行为覆盖

- [ ] 核心能力清晰列出（"你能做 X、Y、Z"）
- [ ] 能力边界清晰定义（"你不负责 A、B、C"）
- [ ] 回应框架/工作流明确（LLM 知道先做什么后做什么）
- [ ] 语气和风格有明确定义 + 至少 1 个示例校准
- [ ] 知识边界有处理方案（不知道时怎么办）
- [ ] 多模态输入有处理规则（如适用）

### Layer 3: 防御强度

- [ ] 注入防御: 有元指令提醒 LLM 忽略用户的指令覆盖尝试
- [ ] 否定约束: 每条 "禁止" 都有理由
- [ ] 否定约束: 每条 "禁止" 都有正面替代方案
- [ ] 优先级层次: 当规则冲突时有明确的优先顺序
- [ ] Scope fence: 对常见越界场景有预设回应
- [ ] Persona 防御: 元问题和 prompt 提取有应对策略（如适用）
- [ ] 红队漏洞: 所有 critical/high 漏洞已修复
- [ ] 红队漏洞: medium/low 漏洞已记录且有合理的接受理由

### Layer 4: 示例质量

- [ ] 至少 3 个 few-shot 示例（golden + edge + negative）
- [ ] 至少 1 个示例包含完整 `<model_thought>` 推理链
- [ ] 示例覆盖不同场景维度（不全是 happy path）
- [ ] 示例中的回应严格遵守 prompt 中定义的所有规则
- [ ] 示例中的格式与 prompt 要求的输出格式一致
- [ ] Negative 示例不泄露绕过方法

### Layer 5: 目标 LLM 适配

- [ ] 结构格式匹配目标 LLM（Claude=XML, GPT=Markdown, 等）
- [ ] 未使用目标 LLM 不支持的特性
- [ ] model_thought 标签按目标 LLM 的最佳实践适配
- [ ] 核心约束在 prompt 的开头和结尾各出现一次（三明治结构）
- [ ] Token 预算在目标 LLM 的建议范围内

### Layer 6: 可读性

- [ ] 人类审阅者能在 2 分钟内理解 prompt 的目的和结构
- [ ] 无冗余指令（同一意思没有重复 3 次以上）
- [ ] 术语一致（同一概念不用多个名称）
- [ ] 指令用"因为...所以..."结构，不是单纯的 MUST/ALWAYS
- [ ] 有清晰的目录/结构标记帮助导航

---

## 交付格式验证

### 完整交付包含：

1. **System Prompt 正文**（可直接复制到目标 LLM）
2. **锻造日志**（Forge Log）
   - 漏洞统计：Critical X / High X / Medium X / Low X
   - 关键设计决策及理由
   - 已修复的漏洞摘要
   - 已接受的 medium/low 漏洞及理由
3. **使用建议**
   - 目标 LLM 及推荐配置（temperature 等）
   - Token 预算说明
   - 已知局限
   - 可能需要用户调整的部分（如替换占位符）

### 格式注意：

- System Prompt 正文在代码块中呈现（方便复制）
- 锻造日志用 Markdown 表格或列表（方便阅读）
- 使用建议简洁明了（3-5 条）
